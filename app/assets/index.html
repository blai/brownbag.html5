  • Performance
  • Options (this is an old one)

Application Frameworks

  • Highly extensible and Light weight
  • Minimal DOM dependency
  • Separation of concerns
  • Active development
  • Options
    • Ember.js (SproutCore 2)
    • Knockout.js
    • Spine.js
    • ExtJs
    • YUI 3

Application Frameworks (cont.)

  • Ember.js
    • SproutCore 1 is used by Apple to build iCloud, but SproutCore 1 and Ember are 2 animals now after some historical change
    • In short: Ember is like an opinionated version of Backbone.js. In general this means you will end up with similar coding structure, with less coding on the Ember version on some common task, and more coding to hack the Ember version for complex cases.

Application Frameworks (cont.)

  • Knockout.js
    • Best documentation and tutorial of all
    • Smaller learning curve, really just need to learn one thing: "data-bind"
    • Concentration of concerns (as oppose to separation of concerns), you basically piled up all connection logic in "data-bind"
    • Perfect fit for one man / contracting projects
    • Performance concern caused by data-bind, data-bind could become maintenance nightmare for large project over time

Application Frameworks (cont.)

  • ExtJs
    • More traction to giant enterprise customers (like HP) as it comes with: tech support, training, commercial licensing (@Alex, are you sold? :P)
    • Comes with many pre-built components (including charting)
  • YUI
    • Y for Yahoo!
    • While YUI may not have the best reputation of all, its one-stop-shop style to solve your javascript/css/html problem as a whole does deserve a thumb up
    • A nice framework extending from YUI3

Application Frameworks (cont.)

  • Spine.js
    • Built using CoffeeScript (small code base)
    • MVC architecture, very similar to Backbone in many ways, with the addtion of Controller
    • Built in localStorage (so you can build an app without a backend at all, good feature for mobile app, though framework such as Backbone has plugin to do the same)
    • Choosing this is a matter of taste IMO
  • Not enough? Here's a bigger list

Application Frameworks (cont.)

  • Backbone.js
    • Extremely flexible / extensible
    • Concept of Collection (on top of Model in MVC) to handle object sets
    • Provides a foundation but does not have opinion on how you should do things.
    • Proven by many large projects

Application Frameworks (cont.)

  • Backbone.js (cont.)
    • Promotes separation of concern, resulting in fragmented file structure, so use of an extension as application framework is recommended:
    • Lumbar/Thorax and its motivation
    • Chaplin
    • Marionette

Application Frameworks (cont.)

  • My pick: Bacokbone (strong buy)
    • Very light weight
    • Event driven
    • Concept of mixin
    • Deeplink routing support

CSS Preprocessor

  • Feature: they pretty much do the same thing
  • Error reporting: Do I need to tell you how frustrated it is to dig out a css error that fails silently in browser
  • Options
    • Sass
    • LESS (I think this is the oldest one)
    • Stylus
  • My pick: Stylus wins by a hair by outstanding in Error reporting (strong buy)

JavaScript Preprocessor

  • CoffeeScript (pros)
    • Clean / readable code, do more with less typing (10 - 40% code reduction)
    • Help avoid common JavaScript pitfalls (e.g. Avoid use of "with" block)
    • Emerging community (and very active)
    • Close to 1 : 1 translation to JavaScript (by the way, the generated script is fairly readable)
    • Close to 1 : 1 performance, for average dev, it actually perform better
    • Proven to work across browser

JavaScript Preprocessor (cont.)

  • CoffeeScript (cons)
    • Steeper learning curve (for those who already know JavaScript, but who does not know JavaScript???)
    • Does not follow any standard specification (e.g. JavaScript follows ECMAScript spec).
    • Pitfalls
    • Too many variations of coding style! (make it hard to do team work). Solution is to follow a code style guide

JavaScript Preprocessor (cont.)

  • My pick: use CoffeeScipt (buy to strong buy)
    • It really took me less than 1 hour to learn
    • Those hard to debug case happened just a few times, and never show up again once I know why it happens
    • Since CoffeeScript compiles to JavaScript, you can always drop it even if you started with it. But this does not happen the other way around
    • If you drink coffee, you might as well pick the cooler one
